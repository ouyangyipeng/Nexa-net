## 1. 项目愿景

### 1.1 使命

**为自治智能体构建去中心化、非侵入式的机器间通讯基础设施。**

### 1.2 核心目标

| 目标 | 描述 | 衡量指标 |
|------|------|----------|
| **无缝集成** | Agent 无需重写核心逻辑即可接入 | 接入时间 < 5 分钟 |
| **高性能** | 支持大规模 Agent 社区的高频通讯 | 路由延迟 < 100ms |
| **安全可信** | 基于密码学的零信任安全架构 | 0 安全事故 |
| **经济可持续** | 内置微交易机制激励服务提供者 | 服务可用性 > 99% |

### 1.3 设计哲学

```
┌─────────────────────────────────────────────────────────────┐
│                    Design Philosophy                        │
│                                                             │
│  1. 非侵入式 (Non-invasive)                                 │
│     - Sidecar Proxy 模式                                    │
│     - Agent 只需调用本地工具                                │
│                                                             │
│  2. 语义优先 (Semantic-First)                               │
│     - 意图驱动而非 URL 驱动                                 │
│     - 自动匹配最合适的服务                                  │
│                                                             │
│  3. 零信任安全 (Zero-Trust Security)                        │
│     - 每次通信都验证                                        │
│     - 基于密码学的身份认证                                  │
│                                                             │
│  4. 原生经济 (Native Economy)                               │
│     - 内置微交易                                            │
│     - 激励对齐                                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 4. 技术决策记录 (ADR)

### ADR-001: 选择 Sidecar Proxy 模式

**状态：** ✅ 已采纳

**日期：** 2026-03-15

**背景：** 需要支持多种异构 Agent 框架（LangChain、AutoGen、自定义脚本），不能要求开发者重写核心逻辑。

**决策：** 采用 Sidecar Proxy 模式，Agent 通过本地接口与 Proxy 通信。

**理由：**
- 非侵入式，现有 Agent 无需修改
- 语言无关，任何语言都可以调用本地接口
- 关注点分离，网络逻辑与业务逻辑解耦

**后果：**
- ✅ 开发者体验好
- ✅ 支持任意 Agent 框架
- ⚠️ 需要额外部署 Proxy 进程
- ⚠️ 本地通信有轻微延迟（< 1ms）

---

### ADR-002: 选择二进制协议而非 JSON

**状态：** ✅ 已采纳

**日期：** 2026-03-16

**背景：** Agent 间通讯需要传输大量结构化数据（长文本、图像矩阵等），JSON 文本格式效率低。

**决策：** 使用 Protobuf/FlatBuffers 作为序列化格式，基于 HTTP/2 或 QUIC 传输。

**理由：**
- 体积缩小 60%-80%
- 序列化/反序列化速度提升 10x+
- 强类型，减少运行时错误

**后果：**
- ✅ 高性能传输
- ✅ 节省带宽和 Token
- ⚠️ 调试不如 JSON 直观
- ⚠️ 需要 Schema 预定义

---

### ADR-003: 选择联邦制拓扑而非纯 P2P

**状态：** ✅ 已采纳

**日期：** 2026-03-17

**背景：** 100 节点规模的 Agent 社区需要高效的语义路由，纯 P2P 的 DHT 查询延迟不可接受。

**决策：** 采用带超级节点的联邦制拓扑，Supernode 维护全局语义索引。

**理由：**
- 路由查询延迟 < 50ms
- 简化 NAT 穿透问题
- 便于监控和管理

**后果：**
- ✅ 高效路由
- ✅ 简化运维
- ⚠️ Supernode 是中心化点（通过多副本缓解）
- ⚠️ 需要运维 Supernode 集群

---

### ADR-004: 选择状态通道而非链上结算

**状态：** ✅ 已采纳

**日期：** 2026-03-18

**背景：** Agent 间可能产生高频微交易（每秒数千次），直接链上结算延迟高、成本大。

**决策：** 采用 Layer 2 状态通道，仅在通道开启/关闭时与主链交互。

**理由：**
- 支持 10 万+ TPS
- 结算成本接近零
- 实时结算体验

**后果：**
- ✅ 高频微交易可行
- ✅ 低延迟结算
- ⚠️ 需要锁定保证金
- ⚠️ 通道管理复杂度增加

---

### ADR-005: 选择 Ed25519 作为默认签名算法

**状态：** ✅ 已采纳

**日期：** 2026-03-19

**背景：** 需要选择一种安全、高效的签名算法用于 DID 和消息签名。

**决策：** 选择 Ed25519 作为默认签名算法，同时支持 Secp256k1 以兼容以太坊生态。

**理由：**
- Ed25519 签名速度快
- 密钥和签名尺寸小
- 安全性经过广泛验证

**后果：**
- ✅ 高性能签名验证
- ✅ 兼容主流 DID 规范
- ⚠️ 需要额外支持 Secp256k1 以兼容以太坊

---

### ADR-006: 选择 HNSW 作为向量索引算法

**状态：** ✅ 已采纳

**日期：** 2026-03-20

**背景：** 语义路由需要高效的向量相似度搜索。

**决策：** 选择 HNSW (Hierarchical Navigable Small World) 作为向量索引算法。

**理由：**
- 查询速度快（O(log n)）
- 召回率高
- 支持动态插入

**后果：**
- ✅ 高效语义搜索
- ✅ 支持实时更新
- ⚠️ 内存占用较高
- ⚠️ 构建索引需要时间

### 2026-04-16 - Phase 10: 集成测试与 E2E HTTP 测试

**变更类型：** 新增 + 修复

**变更内容：**
- 实现 `TestProxy` — 基于 axum 的 HTTP 测试服务器（随机端口 + graceful shutdown via oneshot）
- 实现 `MockNetwork` — 模拟网络拓扑（可控节点可用性状态）
- 创建 5 个 HTTP 级 E2E 测试（`tests/e2e_http_test.rs`）
  - 双 Agent 通信（discover → channel → receipt）
  - 多 Agent 社区（5 agents, cross-discovery, budget control）
  - 经济闭环（10 calls → 10 receipts → settlement）
  - 故障恢复（unavailable agent → fallback routing）
  - 安全验证（unsigned receipt, forged VC, budget exceeded, rate limit）
- 删除重复测试 `tests/integration/e2e_test.rs`（旧版 4 个基础测试副本）
- **修复 ProxyState 关键 bug**: registry 和 router 现在共享同一个 `Arc<RwLock<CapabilityRegistry>` 实例，确保 `/v1/register` 写入对 `/v1/discover` 可见
- **重构 SemanticRouter**: 改用 `Arc<RwLock<CapabilityRegistry/NodeStatusManager>>` 支持并发读写
- **修复 REST handler**: `/v1/discover` 现在传递 `max_candidates` 参数而非使用默认 0

**变更原因：** 原有测试全部是内存级（直接调用 Rust API），未经过 HTTP 层；ProxyState 中 registry/router 使用不同实例导致 REST API 写入不可见

**影响范围：** `src/discovery/router.rs`, `src/proxy/server.rs`, `src/api/rest.rs`, `tests/common/mod.rs`, `tests/e2e_http_test.rs` (新增)

## 6. 贡献指南

### 6.1 如何贡献

1. **Fork 仓库**
   ```bash
   git clone https://github.com/nexa-net/nexa-net.git
   cd nexa-net
   ```

2. **创建分支**
   ```bash
   git checkout -b feature/your-feature-name
   ```

3. **提交变更**
   ```bash
   git add .
   git commit -m ":sparkles: feat: add your feature"
   ```

4. **推送并创建 PR**
   ```bash
   git push origin feature/your-feature-name
   ```

### 6.2 提交规范

遵循 [Conventional Commits](https://www.conventionalcommits.org/) 规范：

```
:<gitmoji>: <type>(<scope>): <subject>

示例：
:sparkles: feat(identity): add DID rotation support
:bug: fix(transport): resolve connection timeout issue
:recycle: refactor(economy): optimize channel management
:memo: docs(readme): update installation guide
```

### 6.3 代码审查标准

| 标准 | 描述 |
|------|------|
| **功能正确** | 代码实现符合设计要求 |
| **测试覆盖** | 核心逻辑有单元测试覆盖 |
| **文档同步** | 相关文档已更新 |
| **风格一致** | 遵循项目编码规范 |
| **安全合规** | 无安全漏洞引入 |
